# xjskp 桌面脚本应用设计文档

## 1. 背景与目标

本项目用于支持微信小程序游戏“向僵尸开炮”的自动化脚本执行。应用形态为可安装的桌面程序，安装后用户可直接启动，应用会在本地启动 Python 服务并自动打开浏览器展示操作界面。

本次设计遵循以下已确认约束：

- 开发流程采用**方案 2（严格串行）**。
- MVP 优先目标：先完成“应用壳子可安装可启动可观测”。
- MVP 演示脚本数量：1 个。
- MVP 演示脚本类型：界面识别 + 条件判断 + 分支。
- 桌面方案：`PyInstaller + 本地 Python 服务 + 自动打开系统浏览器`。
- MVP 平台范围：仅 `macOS`。
- 进度跟踪粒度：里程碑级。

## 2. 设计原则

- 先交付可运行壳子，再接入脚本能力，避免双线并发带来的管理复杂度。
- 模块边界清晰，界面层不直接依赖自动化实现细节。
- 日志优先，所有关键步骤可观测、可复盘。
- 保持最小可行（YAGNI），MVP 仅覆盖 1 个真实演示任务。
- 规划文档作为“可中断/可继续”的唯一进度锚点，持续更新同一文件。

## 3. 总体架构（串行阶段）

### 阶段 1：桌面壳子与运行闭环

目标是先完成“可安装、可启动、可观察”。

- 打包产物：`macOS .dmg`（后续再扩 `Windows .exe`）。
- 启动行为：双击应用后拉起本地 Python 服务，并自动打开浏览器到本地地址。
- 最简界面：
  - 任务名称与勾选框
  - 执行按钮
  - 日志显示区域（实时输出步骤与结果）
- 进程管理：
  - 应用退出时优雅停止本地服务
  - 异常时保留关键日志

阶段 1 完成定义：

- 安装成功、启动成功、浏览器自动打开成功。
- 页面可交互，日志可实时显示。
- 打包后的应用可重复运行。

### 阶段 2：接入脚本能力（1 个带分支演示任务）

在阶段 1 稳定后接入真实脚本执行：

- 任务注册与调度入口。
- 最小执行引擎（顺序步骤 + 条件判断 + 分支）。
- 复用原子能力接口（窗口抓取、识别定位、动作执行）。
- 1 个真实演示任务，至少包含一次条件分支。

阶段 2 完成定义：

- 可从界面勾选并执行该任务。
- 任务可稳定跑通分支流程。
- 日志支持问题复盘（步骤、结果、分支原因、错误）。

### 阶段 3：可持续迭代能力

围绕“随时中断/继续”建立治理机制：

- 固化里程碑状态记录（未开始/进行中/完成/风险）。
- 每次迭代记录下一步目标、风险和结果。
- 形成新增任务接入规范，降低后续扩展成本。

## 4. 模块划分

### Desktop Launcher（桌面启动器）

- 职责：应用启动、服务进程管理、浏览器自动打开、退出清理。
- 不负责：脚本执行逻辑。

### Local Service（Python 本地服务）

- 职责：提供 HTTP API、托管页面资源、维护任务状态与日志流。
- 阶段策略：阶段 1 使用占位执行；阶段 2 切换真实任务执行。

### Web UI（浏览器界面）

- 职责：展示任务列表、勾选状态、执行入口和日志面板。
- 边界：仅通过 API 通信，不直接触达系统自动化能力。

### Task Runtime（任务运行时，阶段 2 启用）

- 职责：任务定义、步骤执行、分支决策、结果汇总。
- 对外接口：统一“执行任务”入口，供 `service` 调用。

### Automation Adapters（自动化适配层，阶段 2 启用）

- 职责：封装窗口抓取、图像识别、点击输入等原子能力。
- 边界：隔离平台差异，MVP 仅实现 macOS。

### Logging & Diagnostics（日志诊断）

- 职责：结构化日志协议、实时流转、落盘持久化。
- 目的：同时支持 UI 观察与离线问题定位。

## 5. 建议目录结构

```text
xjskp/
  docs/
    superpowers/specs/
  src/
    launcher/
    service/
    web/
    runtime/
    adapters/
    shared/
  scripts/
    build-macos.sh
    run-dev.sh
  artifacts/
```

说明：

- `runtime/` 与 `adapters/` 可在阶段 1 先建空骨架，阶段 2 再填充实现。
- `artifacts/` 存放打包输出，建议加入 `.gitignore`。

## 6. 里程碑开发流程（里程碑级）

### M0：项目基线搭建

目标：建立可持续开发骨架。

产出：

- 目录结构初始化（`launcher/service/web/shared`）。
- 本地一键启动脚本。
- 基础日志通道（终端 + 文件）。

完成定义：

- 新环境可一键启动并打开页面（允许占位内容）。
- 日志链路可用。

### M1：桌面壳子闭环（阶段 1 核心）

目标：可安装、可启动、可查看日志。

产出：

- `PyInstaller` 打包流程与 `dmg` 产物。
- 启动自动拉起服务并打开浏览器。
- UI：任务勾选、执行按钮、实时日志区域。
- 退出清理机制。

完成定义：

- 可生成并安装 `dmg`。
- 双击启动后可进入 UI。
- 占位任务执行时日志完整可见。
- 异常可在日志中定位失败阶段。

### M2：脚本能力最小闭环（阶段 2 核心）

目标：接入 1 个真实分支任务。

产出：

- 任务定义与注册机制。
- 步骤执行器与条件分支逻辑。
- 1 个“识别 + 判断 + 分支”的演示任务。

完成定义：

- 从 UI 可触发并跑通任务。
- 分支路径可解释（日志体现触发依据）。
- 失败路径可定位并可重试。

### M3：迭代机制固化（阶段 3）

目标：支持后续稳定扩展新任务。

产出：

- 里程碑状态模板（状态、风险、下一步）。
- 新任务接入规范。
- 变更记录区（持续更新）。

完成定义：

- 任意时点可明确当前所处里程碑。
- 可按文档快速恢复开发上下文。

## 7. 日志、错误处理与诊断策略

- 启动失败：提供明确错误提示并记录 root cause。
- 任务失败：标记任务状态，保留步骤级上下文。
- 异常退出：保留最后一次运行摘要，便于下次启动排查。
- 日志结构建议字段：`timestamp`、`level`、`task_id`、`step`、`message`、`result`、`error_code`。

## 8. 测试与验收策略

- M1 冒烟：安装、启动、UI 打开、日志链路、退出清理。
- M2 功能：主路径、分支路径、异常路径、重试路径。
- M3 回归：新增任务不破坏既有任务与壳子流程。

## 9. 进度管理与中断恢复机制

本文件即项目流程主文档。后续每次推进时只更新以下区域：

- 当前里程碑状态（`M0/M1/M2/M3`）。
- 本次完成内容（Done）。
- 风险与阻塞（Risks/Blockers）。
- 下一步（Next）。

推荐状态模板：

```text
Current Milestone: Mx (Not Started / In Progress / Done)
Done:
- ...
Risks/Blockers:
- ...
Next:
- ...
```

## 10. 非目标（MVP 不做）

- 不在 MVP 同步支持 macOS 与 Windows 自动化能力。
- 不在 MVP 一次性交付多个复杂脚本任务。
- 不在 MVP 追求高保真 UI 设计，仅保证可用与可观测。

- 每一步可独立中断并恢复。
